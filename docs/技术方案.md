# 最简单可行技术方案

## 1. 方案结论

本项目首期推荐采用：**`WireGuard` 星型组网 + Windows 最小 CLI 客户端**。

这是当前最简单、最稳定、最容易落地的方案，原因如下：

- 不自研隧道协议，不自己处理底层加密与虚拟网卡。
- 不引入账号系统、控制面、复杂后台。
- 直接复用现成服务器 `47.119.136.200` 作为联机中转节点。
- 客户端只做最小包装：连接、断开、状态显示、错误提示。

首期目标不是“全功能加速器”，而是**让两台或多台客户端接入同一个虚拟网络，并通过虚拟 IP 完成游戏联机**。

## 2. 为什么选择这个方案

相较于其他方案：

- 自研 VPN / 自研隧道：实现成本高，不符合“最简单可行”。
- `Headscale` / `Tailscale` 自建：需要额外控制面，复杂度更高。
- `ZeroTier` 自建控制器：联机可行，但部署和控制面仍比 `WireGuard` 更复杂。
- `SoftEther` 二层桥接：虽然更接近传统局域网，但部署、桥接和调试复杂度明显更高。

因此，首期选择 `WireGuard` 更合适。

## 3. 联机方式定义

首期联机方式定义如下：

- 客户端连接到 `47.119.136.200` 上的 `WireGuard` 服务端。
- 每个客户端获得一个固定虚拟 IP，例如 `10.99.0.x`。
- 多个客户端通过服务器进行三层互通。
- 游戏优先通过“**虚拟 IP 直连**”方式联机。

这意味着首期**不保证局域网广播发现**，即：

- 某些依赖局域网广播找房间的游戏，可能无法自动发现房主。
- 这类游戏在首期方案中优先采用“输入房主虚拟 IP 直连”的方式。

这个取舍可以显著降低实现复杂度，同时满足“最小可用联机”的核心目标。

## 4. 系统架构

### 4.1 服务端

服务器 `47.119.136.200` 负责：

- 运行 `WireGuard`。
- 监听固定 UDP 端口，例如 `51820/udp`。
- 为客户端提供虚拟网络接入。
- 在客户端之间转发 VPN 子网流量。
- 维护客户端配置。

建议服务端虚拟网段：

- VPN 网段：`10.99.0.0/24`
- 服务端地址：`10.99.0.1/24`

### 4.2 客户端

客户端首期采用 **Windows CLI 程序**，职责如下：

- 读取本地配置。
- 使用预置服务器地址 `47.119.136.200` 发起连接。
- 调用本机已安装的官方 `WireGuard for Windows` 完成隧道启停。
- 输出最小状态：未连接 / 连接中 / 已连接 / 连接失败。
- 提供最基本错误信息。

## 5. 客户端最小实现边界

客户端只实现以下能力：

- `connect`：启动 VPN 连接。
- `disconnect`：断开 VPN 连接。
- `status`：查看当前连接状态。
- `config`：读取默认服务端配置。
- 基本错误提示，例如：
  - 未安装 `WireGuard`
  - 配置文件不存在
  - 权限不足
  - 无法连通服务器

首期**不实现**：

- GUI 桌面界面
- 用户注册登录
- 自动分配客户端配置
- 远程控制面 API
- 节点切换
- 智能选路
- 流量统计
- 自动更新

## 6. 配置与发放方式

为了避免引入控制面系统，首期采用**人工发放客户端配置**：

- 服务端管理员在服务器上生成每个客户端独立的 `WireGuard` 配置。
- 每个客户端拿到自己的 `.conf` 文件。
- 客户端程序读取该配置并控制连接。

这样做的好处是：

- 不需要注册接口。
- 不需要鉴权系统。
- 不需要数据库。
- 开发量最小。

## 7. 客户端技术选型

首期推荐客户端技术栈：**Go CLI**。

选择原因：

- 可编译成单个 Windows 可执行文件。
- 适合调用系统命令与检查服务状态。
- 发布简单，依赖少。
- 实现连接、断开、状态检查足够直接。

推荐目录职责：

- `cmd/client`：CLI 入口
- `internal/config`：读取配置
- `internal/wg`：调用 `WireGuard` 命令
- `internal/status`：状态检测

## 8. 服务端实现方式

服务端推荐使用 Linux（如 Ubuntu），并通过 SSH 在 `47.119.136.200` 上完成操作。

服务端首期仅需要：

- 安装 `WireGuard`
- 开启 IP 转发
- 放行 `51820/udp`
- 创建 `wg0.conf`
- 为每个客户端追加 peer 配置

首期服务端不需要：

- Web 管理后台
- 数据库
- 用户体系
- 自动签发 API

## 9. 连接流程

### 9.1 首次准备

1. 服务器部署 `WireGuard`。
2. 管理员为每个客户端生成密钥与配置。
3. 将客户端配置文件下发给用户。
4. 用户机器预先安装 `WireGuard for Windows`。
5. 用户运行最小客户端。

### 9.2 建连流程

1. 客户端读取本地 `.conf`。
2. 客户端调用本机 `WireGuard` 启动隧道。
3. 客户端检测是否获得虚拟 IP。
4. 客户端检测是否可以访问 `10.99.0.1`。
5. 成功后显示“已连接”。

### 9.3 联机流程

1. 房主客户端连接 VPN。
2. 玩家客户端连接 VPN。
3. 各客户端通过虚拟 IP 互通。
4. 游戏使用房主的虚拟 IP 进行直连联机。

## 10. 状态检测方案

客户端状态可按以下最小规则判断：

- 未连接：隧道未启动。
- 连接中：已发起启动但尚未确认连通。
- 已连接：隧道存在且可访问 `10.99.0.1`。
- 连接失败：隧道启动失败，或服务端不可达。

建议检测方式：

- 检查 `WireGuard` 隧道服务是否存在。
- 检查虚拟网卡或隧道状态。
- `ping` 服务端虚拟 IP `10.99.0.1`。

## 11. 最小交付物

首期最小交付物建议如下：

- 一份服务端部署脚本或部署文档。
- 一份客户端配置模板。
- 一个 Windows CLI 客户端可执行文件。
- 一份使用说明。
- 一份联机验证记录。

## 12. 风险与边界

### 12.1 已知边界

- 首期不处理广播找房间问题。
- 首期默认只支持 Windows 客户端。
- 首期依赖用户本机已安装 `WireGuard for Windows`。
- 启动隧道可能需要管理员权限。

### 12.2 风险控制

为保持简单，首期采取以下策略：

- 不嵌入 `WireGuard` 驱动，只调用官方客户端。
- 不做自动注册和自动配网。
- 不做复杂 UI。
- 不做跨平台支持。

## 13. 验收标准

当满足以下条件时，可认为技术方案落地成功：

- 服务端在 `47.119.136.200` 上成功运行。
- 两台 Windows 客户端可分别连接该服务器。
- 两台客户端在 VPN 内可互相 `ping` 通。
- 游戏可通过虚拟 IP 完成联机验证。
- 用户操作保持最小化，不包含无关功能。

## 14. 第二阶段可选扩展

如果首期验证通过，后续再考虑：

- 简单 GUI 包装
- 自动导入配置
- 广播转发 / 房间发现支持
- 配置签发接口
- 多节点与测速

但这些内容都**不属于当前最小可行方案**。
