# 最小可用游戏联机客户端需求文档

## 1. 项目目标

实现一个**最简单、可实际使用**的游戏联机客户端，用于让用户通过已有服务器 `47.119.136.200` 建立联机通道，从而支持游戏联机。

本项目当前目标不是完整商业化“游戏加速器”，而是优先实现“**能连上、能联机、可验证**”的最小可用版本。

## 2. 设计原则

- 只实现联机所必需的能力，不添加不必要功能。
- 优先保证方案简单、可部署、可调试。
- 客户端功能尽量少，避免复杂 UI 和额外系统。
- 服务端已有可用机器，优先复用 `47.119.136.200`。

## 3. 核心需求

### 3.1 客户端目标

客户端至少需要完成以下能力：

- 能让用户在本机上启动联机连接。
- 能与服务器建立通信。
- 能将本机与远端联机网络打通，使游戏可以通过该通道进行联机。
- 能提供最基本的连接状态反馈，便于用户判断是否连接成功。

### 3.2 最小功能范围

本阶段客户端仅包含以下最小功能：

- 服务器地址配置（默认使用 `47.119.136.200`）。
- 一键连接。
- 一键断开。
- 显示当前连接状态，例如“未连接 / 连接中 / 已连接 / 连接失败”。
- 基本错误提示，便于定位连接失败原因。

### 3.3 联机实现要求

最小方案的重点是“实现联机”，因此方案上应优先选择：

- 成熟、稳定、实现成本低的组网或隧道方式。
- 客户端部署简单，用户操作步骤少。
- 服务端可以通过 SSH 在 `47.119.136.200` 上完成部署与配置。

## 4. 非目标范围

以下内容明确**不在当前阶段实现范围内**：

- 账号注册、登录、用户体系。
- 多节点切换。
- 智能选路、测速、延迟排行。
- 流量统计、套餐、订阅、支付。
- 自动更新。
- 复杂桌面 UI。
- 多平台深度适配。
- 后台管理系统。
- 与“加速”无关的附加工具功能。

## 5. 角色划分

### 5.1 客户端

客户端负责：

- 接收用户连接操作。
- 按预设配置连接服务器。
- 在本机建立联机所需的网络接入。
- 向用户展示最小必要状态。

### 5.2 服务端

服务端负责：

- 接收客户端接入。
- 提供联机网络的中转或组网能力。
- 为多个客户端提供互通基础。
- 支持通过 SSH 方式进行部署、配置和维护。

## 6. 用户使用流程

最小可用版本的预期流程如下：

1. 用户启动客户端。
2. 客户端默认读取服务器地址 `47.119.136.200`。
3. 用户点击连接。
4. 客户端与服务器建立联机通道。
5. 客户端显示“已连接”。
6. 用户启动或进入游戏联机。
7. 如需退出，用户点击断开。

## 7. 成功标准

满足以下条件即可认为本阶段目标完成：

- 客户端可正常启动。
- 客户端可连接到 `47.119.136.200`。
- 至少两台客户端能通过该方案进入同一联机网络。
- 用户可通过该网络进行游戏联机验证。
- 整体使用步骤简单，无明显多余功能。

## 8. 后续实现方向

在正式实现阶段，应继续细化以下内容：

- 选定最简单可行的联机网络方案。
- 明确客户端技术栈与运行方式。
- 明确服务端部署方式。
- 明确连接状态检测与错误提示方式。
- 完成最小原型并进行联机验证。
